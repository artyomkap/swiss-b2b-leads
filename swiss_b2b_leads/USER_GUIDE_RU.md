# Swiss B2B Lead Search - Руководство пользователя

### Что Делает Инструмент

Swiss B2B Lead Search собирает B2B-лиды по Швейцарии из нескольких источников, сравнивает качество источников, по возможности обогащает данные с сайтов компаний, сохраняет историю поисков и экспортирует результаты в Excel.

Инструмент нужен для тестирования: какие источники, регионы, города и категории дают лучшие данные по нужной отрасли.

### Обзор Меню

#### Верхняя Панель

- **Swiss B2B Lead Search**: главная страница поиска.
- **History**: страница истории поисков. Там можно смотреть прошлые поиски, продолжать прерванные, удалять, объединять и экспортировать результаты.

#### API Keys

В этом блоке настраиваются внешние API-интеграции.

- **Google Places**: поиск компаний и мест через Google Places.
- **SerpAPI**: Google-подобный web search через SerpAPI.
- **Tavily**: альтернативный web search, если не используется SerpAPI.
- **Firecrawl**: дополнительное глубокое обогащение сайтов.
- **Eye button**: показать или скрыть ключ при редактировании.
- **Save & Apply Keys**: сохранить ключи и применить их без перезапуска приложения.

Маленькие статусные точки показывают состояние источников: доступен, отсутствует ключ или есть недавняя API-проблема, например quota/rate limit.

#### Search Parameters

Здесь настраивается сам поиск.

##### Locations

Строка поиска локаций принимает:

- города, например `Zurich`, `Basel`, `Lugano`;
- кантоны, например `ZH`, `Canton Zurich`, `Ticino`;
- регионы, например `Romandie`, `Central Switzerland`, `Ticino`;
- кастомные зоны, введенные вручную.

Выбранные локации отображаются ниже как chips. Чтобы удалить локацию, нажмите `x` на chip.

Режимы покрытия:

- **Top cities**: рекомендуемый режим по умолчанию. Кантон или регион раскрывается в основные города. Это дает нормальное покрытие без слишком большого количества API-запросов.
- **Area query**: поиск по названию зоны напрямую, без раскрытия в города.
- **All cities**: раскрывает кантон или регион во все города, которые сейчас есть во внутреннем справочнике. Это шире, но может быть дороже и дольше.

Важно: если выбран регион, это не означает автоматический поиск по всем муниципалитетам и районам Швейцарии. Регион раскрывается в доступный список городов/локаций внутри этого региона.

##### Industries

Здесь выбираются целевые отрасли.

Кнопки:

- **All**: выбрать все отрасли.
- **None**: снять выбор со всех отраслей.
- **Custom search term**: добавить свой поисковый термин.
- **+**: добавить введенный кастомный термин.

Примеры:

- `Banks`
- `Insurance companies`
- `Shopping centers`
- `Property management companies`
- `Garages / Car dealerships`

##### Target Qualifying Leads

Настраивает, сколько финальных подходящих лидов система должна попытаться найти.

Например, если указано `30`, система будет искать до 30 подходящих лидов или остановится, когда достигнет максимального количества раундов.

##### Quality Requirements

Дополнительные фильтры для финального списка лидов:

- **Phone number required**: оставить только лиды с телефоном.
- **Email address required**: оставить только лиды с email.
- **Website required**: оставить только лиды с сайтом.
- **Clear**: сбросить все требования качества.

Чем строже требования, тем меньше обычно получается финальных лидов.

##### Max Fetch Rounds

Количество раундов сбора данных.

Если цель не достигнута в первом раунде, следующий раунд увеличивает размер сбора. Больше раундов может дать больше результатов, но поиск будет дольше и может использовать больше API-кредитов.

##### Website Enrichment Workers

Количество сайтов, которые обрабатываются параллельно.

- Меньшее значение: медленнее, но стабильнее.
- Большее значение: быстрее, но сайты могут чаще блокировать запросы.

Рекомендуемый диапазон: `5-10`.

#### Sources

Выбор источников данных.

- **search.ch**: швейцарский справочник. Хорошо подходит для компаний и телефонных данных.
- **Google Places**: данные компаний/мест из Google. Требует Google API key.
- **Google Search**: web search через SerpAPI или Tavily. Полезен для поиска сайтов и контактных страниц.
- **Website Parser**: заходит на найденные сайты компаний и пытается извлечь email, телефон и контактную страницу.
- **Firecrawl**: дополнительное расширенное обогащение сайтов. Требует Firecrawl API key.

Если ключ отсутствует или провайдер достиг лимита, источник может быть пропущен, а остальные источники продолжат работать.

#### Run Search

Запускает поиск с выбранными настройками.

Перед запуском система считает estimate:

- количество раскрытых локаций;
- примерное количество source queries;
- примерное количество внешних API-вызовов;
- уровень cost warning.

Если поиск слишком большой, интерфейс может попросить подтверждение перед запуском.

#### Pause / Resume

Во время поиска:

- **Pause**: запросить паузу текущего job.
- **Resume**: продолжить paused job.

Если сервер был перезапущен, interrupted search можно продолжить со страницы History.

### Правая Часть Интерфейса

#### Progress

Live-лог выполнения поиска:

- текущий раунд;
- активный источник;
- текущая локация и категория;
- сколько записей собрано;
- прогресс обогащения сайтов;
- API warnings или отключенные источники.

Если API возвращает quota, rate limit, billing или invalid key ошибку, уведомление появится здесь.

#### Metrics

Сводные показатели:

- **Raw records**: сырые записи до финальной очистки.
- **Unique leads**: лиды после дедупликации.
- **Qualified**: лиды, которые соответствуют выбранным quality requirements.
- **With phone**: лиды с телефоном.
- **With email**: лиды с email.

#### Source Comparison

Сравнение качества по источникам:

- сколько лидов собрано;
- phone rate;
- email rate;
- средний quality score.

Этот блок помогает понять, какой источник лучше работает для конкретной отрасли и региона.

#### Leads

Финальная таблица лидов.

Поле фильтра ищет по компании, городу, email, телефону и другим видимым полям.

Обычно в таблице есть:

- название компании;
- отрасль;
- город/кантон;
- телефон;
- email;
- сайт;
- источник;
- quality score.

#### Export

Экспорт текущих результатов в Excel:

- **File 1: Target Leads Excel**: экспортирует текущий показанный target-list, например первые 30 лидов, если target равен 30.
- **File 2: All Qualified Excel**: экспортирует все дедуплицированные лиды, которые прошли выбранные quality requirements, без отсечки по target count.

Пояснение в блоке Metrics показывает, сколько source records было собрано, сколько unique leads осталось после дедупликации и сколько qualified leads доступно для File 2.

### History

Открывается через ссылку **History** в верхней панели.

Функции:

- **Back to Search**: вернуться на главную страницу поиска.
- **Select all**: выбрать все записи истории.
- **Merge & Export Selected**: объединить выбранные поиски и экспортировать одним файлом.
- **Delete Selected**: удалить выбранные записи истории.
- **Export**: экспортировать один поиск.
- **Resume**: продолжить interrupted или paused search.
- **Watch Live**: подключиться к уже запущенному поиску.
- **Delete icon**: удалить одну запись истории.

Каждая карточка истории показывает:

- выбранные локации и expanded terms;
- отрасли;
- target count;
- estimated queries;
- статус;
- raw/qualified/final lead count;
- cost warning level;
- API warnings;
- source breakdown.

### Рекомендуемый Рабочий Процесс

1. Откройте приложение.
2. Проверьте или добавьте API keys.
3. Для первого теста выберите одну локацию: город или кантон.
4. Выберите одну или две отрасли.
5. Оставьте coverage mode **Top cities**.
6. Поставьте реалистичный target, например `30-100`.
7. Для широкого теста не включайте quality requirements. Для более строгого результата включите phone/website/email.
8. Выберите источники.
9. Проверьте estimate.
10. Нажмите **Run Search**.
11. Следите за **Progress**.
12. В **Source Comparison** проверьте, какой источник дал лучшие данные.
13. Проверьте таблицу **Leads**.
14. Экспортируйте File 1 для target-list или File 2 для всех qualified leads.
15. Через **History** можно позже продолжить, экспортировать, объединить или удалить поиски.

### API Limits

Система пытается автоматически определить типичные API-проблемы:

- отсутствующий ключ;
- неправильный ключ;
- quota exceeded;
- rate limit;
- billing/credits problems.

Если один провайдер падает по лимитам, поиск должен продолжиться через остальные включенные источники. Проблемный источник отображается в UI и сохраняется в History как API warning.

Для полноценного production-использования нужны активные платные API keys с достаточным количеством лимитов.
